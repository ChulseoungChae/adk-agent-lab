# 4. 제 4장: 공정 이력 추적과 데이터 기반의 레시피 최적화

# 제 4장: 공정 이력 추적과 데이터 기반의 레시피 최적화

반도체 제조 공정에서 발생한 결과의 원인을 파악하기 위해서는 단순한 실시간 데이터뿐만 아니라, 과거의 모든 공정 이력을 체계적으로 추적하는 능력이 필수적입니다. 본 장에서는 플랫폼의 `fetch_equipment_history` 기능을 활용하여 특정 레시피의 수행 이력을 분석하고, 이를 통해 공정의 안정성을 평가하는 방법을 다룹니다.

## 1. 공정 이력 데이터의 구조

플랫폼에 저장된 공정 이력(Equipment History)은 특정 레시피(`process_recipe`)가 실행된 각 작업 단위인 `PJOB`을 기준으로 관리됩니다. 각 이력에는 다음과 같은 핵심 정보가 포함되어 있습니다.

| 필드명 | 설명 |
| :--- | :--- |
| `timestamp` | 공정이 시작되거나 종료된 시각 |
| `process_recipe` | 실행된 공정의 레시피 명칭 (예: `RUN\0_P_IH_HD02`) |
| `status` | 작업의 최종 상태 (성공, 실패, 중단 등) |
| `duration` | 공정 소요 시간 |

이러한 구조화된 데이터는 엔지니어가 특정 시점에 발생한 불량(Yield Loss)의 원인을 찾기 위해 과거의 동일 레시피 실행 이력을 역추적할 때 강력한 도구가 됩니다.

## 2. 사례 연구: 특정 레시피의 실행 이력 분석

실제 플랫폼의 데이터를 조회하여 `RUN\0_P_IH_HD02` 레시피의 실행 이력을 분석해 보겠습니다. 플랫폼의 이력 조회 기능을 통해 확인된 최근 작업 내역은 다음과 같습니다.

**[조회된 공정 이력 예시]**
* **레시피명:** `RUN\0_P_I_HD02-1`
* **최근 수행 기록:**
    * `2026-03-18 15:11:47`: 작업 완료 (정상)
    * `2026-03-18 15:11:08`: 작업 완료 (정상)
    * `2026-03-18 15:10:29`: 작업 완료 (정상)

이와 같이 연속적인 작업 기록을 통해 설비의 가동률(Uptime)과 레시피의 반복 재현성(Repeatability)을 한눈에 파악할 수 있습니다. 만약 특정 시간대에 `status`가 'Error'로 기록되거나 작업 시간이 급격히 변동되었다면, 이는 설비의 물리적 마모나 부품 교체 시점이 다가왔음을 암시하는 중요한 신호입니다.

## 3. 데이터 기반의 레시피 최적화 전략

단순한 이력 조회를 넘어, 축적된 데이터를 활용하면 레시피의 파라미터를 최적화할 수 있습니다.

1. **패턴 분석:** `process_recipe`별로 `duration`의 표준 편차를 계산하여, 공정 산포가 큰 레시피를 식별합니다.
2. **상관관계 도출:** 이상 감지 로그(`anomaly_logs`)와 공정 이력을 결합하여, 어떤 파라기미터의 변동이 특정 레시피의 실패를 유도하는지 분석합니다.
3. **예방 정비(PdM) 연계:** 이력 데이터에서 발견되는 미세한 성능 저하 추세(Trend)를 바탕으로, 설비 정지(Downtime)가 발생하기 전에 사전 정비 계획을 수립합니다.

결론적으로, 공정 이력 데이터는 단순한 기록을 넘어 차세대 스마트 팩토리를 구현하기 위한 핵심적인 자산입니다. 엔지니어는 플랫폼의 이력 추적 기능을 활용하여 '사후 대응'에서 '사전 예방'으로 공정 관리 패러다임을 전환해야 합니다.
